**Step 2 (GREEN): Create src/config.ts**

Create `src/config.ts` with the following structure. Follow the research patterns exactly:

1. Import dotenv, zod, jose:

   import { config as dotenvConfig } from "dotenv";

3. Define and export `envSchema`:
   ```typescript
   export const envSchema = z.object({
     PORT: z.string().default("8000").transform(Number),
     WIKIJS_BASE_URL: z.string().url("WIKIJS_BASE_URL must be a valid URL"),
     WIKIJS_TOKEN: z.string().min(1, "WIKIJS_TOKEN must not be empty"),
     AZURE_TENANT_ID: z.string().uuid("AZURE_TENANT_ID must be a valid UUID"),
     AZURE_CLIENT_ID: z.string().uuid("AZURE_CLIENT_ID must be a valid UUID"),
     MCP_RESOURCE_URL: z.string().url("MCP_RESOURCE_URL must be a valid URL"),
   }).transform((env) => ({
     port: env.PORT,
     wikijs: {
       baseUrl: env.WIKIJS_BASE_URL,
       token: env.WIKIJS_TOKEN,
     },
     azure: {
       tenantId: env.AZURE_TENANT_ID,
       clientId: env.AZURE_CLIENT_ID,
       resourceUrl: env.MCP_RESOURCE_URL,
       jwksUri: `https://login.microsoftonline.com/${env.AZURE_TENANT_ID}/discovery/v2.0/keys`,
       issuer: `https://login.microsoftonline.com/${env.AZURE_TENANT_ID}/v2.0`,
     },
   }));
   ```

4. Export the AppConfig type:
   ```typescript
   export type AppConfig = z.output<typeof envSchema>;
   ```

5. Run safeParse on process.env and handle failure with grouped error output:
   - Separate errors into "missing" (messages containing "Required") and "invalid" (everything else)
   - Print categorized list under `=== Configuration Error ===` header
   - End with `See example.env for required variables.`
   - Call `process.exit(1)` on failure

6. Export the validated config:
   ```typescript
   export const config = result.data;
   ```

7. Initialize and export JWKS resolver (lazy, no network call):
   ```typescript
   export const jwks = createRemoteJWKSet(
     new URL(config.azure.jwksUri)
   );
   ```

8. Create and export a `maskValue` function and a `logConfig` function:
   - `maskValue(value: string, showChars: number = 4): string` -- shows first N chars then "****"
   - `logConfig(cfg: AppConfig): void` -- logs structured config summary with masked sensitive values
   - Port and URLs are not masked; tenant ID and client ID show 8 chars; WIKIJS_TOKEN shows 4 chars
   - Include derived JWKS URI and issuer in the log output

**Important testability design:** The `envSchema` is exported as a named export so tests can call `envSchema.safeParse(testEnv)` directly without triggering the module-level `process.exit(1)`. The module-level validation only runs when the module is first imported in production (server startup). In tests, importing just `envSchema` is safe because Vitest's module resolution can handle selective imports, and the top-level code runs once when the module loads -- tests should use `vi.mock` or dynamic import patterns if the top-level code causes issues.

**Practical approach for testability:** Structure the module so the schema definition and the validation/exit logic are separate:
- Export `envSchema` at the top (before any safeParse calls)
- The module-level safeParse/exit code runs after the schema export
- Tests import only `envSchema` but the module top-level code will still execute on import
- To avoid `process.exit(1)` killing the test runner when testing invalid configs, the test file should NOT import the module at the top level. Instead, use a pattern where `envSchema` is defined in a way tests can access without triggering the exit.

**Recommended pattern:** Create a `loadConfig()` function that does the safeParse + exit + JWKS init, and call it at module level. Export `envSchema` separately before `loadConfig()`. For the test to work without triggering exit, mock `process.exit` using `vi.spyOn(process, "exit").mockImplementation(() => { throw new Error("exit"); })` or simply test `envSchema.safeParse()` directly (which does NOT call process.exit).

Since tests use `envSchema.safeParse(testEnv)` directly (passing a custom object, NOT process.env), the module-level `envSchema.safeParse(process.env)` will also run on import. Ensure the test environment has valid env vars set (or mock them) so the module-level parse doesn't fail during test import. Alternatively, extract `envSchema` into a separate file (`src/env-schema.ts`) and have `config.ts` import from it -- but this adds unnecessary files. The simplest approach: in test setup, set the required env vars before importing.
