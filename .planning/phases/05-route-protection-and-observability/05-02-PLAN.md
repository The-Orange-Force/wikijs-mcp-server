2. Create `src/tool-wrapper.ts` -- Tool invocation timing wrapper (per user decision: "Log tool name and execution duration for each MCP tool invocation"):

   Export a `wrapToolHandler` function that wraps any MCP tool handler to measure and log execution duration:

   ```typescript
   import { requestContext } from "./request-context.js";

   export function wrapToolHandler<TArgs, TResult>(
     toolName: string,
     handler: (args: TArgs) => Promise<TResult>
   ): (args: TArgs) => Promise<TResult> {
     return async (args: TArgs): Promise<TResult> => {
       const ctx = requestContext.getStore();
       const start = performance.now();
       try {
         const result = await handler(args);
         const duration = Math.round(performance.now() - start);
         ctx?.log.info({ toolName, duration, userId: ctx.userId, username: ctx.username }, `Tool invocation: ${toolName}`);
         return result;
       } catch (error) {
         const duration = Math.round(performance.now() - start);
         ctx?.log.error({ toolName, duration, userId: ctx.userId, username: ctx.username, error: String(error) }, `Tool error: ${toolName}`);
         throw error;
       }
     };
   }
   ```

   Key design points:
   - Uses `requestContext.getStore()` to access the AsyncLocalStorage context set by `requestContext.run()` in the POST /mcp handler
   - Uses `performance.now()` for sub-millisecond timing precision, rounds to integer ms for log readability
   - Logs at `info` level on success, `error` level on failure
   - Includes `userId` and `username` explicitly in the log payload (these are also in the child logger bindings from the auth preHandler, but including them explicitly in the tool invocation log ensures they are present even if the logger binding chain is misconfigured)
   - The wrapper is applied at tool registration time in `createMcpServer` (Phase 1). If Phase 1 has not been executed yet, document the integration point as a TODO comment in mcp-routes.ts

3. Create `src/routes/mcp-routes.ts` -- Protected MCP routes as Fastify encapsulated plugin:

   Export an async function `protectedRoutes(fastify: FastifyInstance)` that:

   a. Adds a scoped `preHandler` hook that runs ONLY for routes inside this plugin:
      - Extract Bearer token from `request.headers.authorization` (strip "Bearer " prefix)
      - If no token: call `mapMissingTokenError()` from auth-errors.ts, log at warn level, send error response with correlation_id field, `return reply`
      - If token present: call Phase 4's `validateToken(request)` to validate
      - On validation success: set `request.user`, create child logger with `{ userId: user.oid, username: user.preferred_username }` via `request.log = request.log.child({...})`
      - On validation failure: call `mapJoseErrorToRfc6750()`, log at warn level ("Auth rejected"), log raw error at debug level, send RFC 6750 error response with `{ error, error_description, correlation_id: request.id }` JSON body and WWW-Authenticate header, `return reply`

   b. Registers `POST /mcp` route:
      - Creates StreamableHTTPServerTransport and McpServer per existing Phase 1 pattern
      - Wraps `transport.handleRequest()` in `requestContext.run()` with context containing `{ correlationId: request.id, userId: request.user?.oid, username: request.user?.preferred_username, log: request.log }`
      - This ensures tool handlers can access correlation ID and user identity via `requestContext.getStore()`
      - When registering tools on the McpServer, wrap each tool handler with `wrapToolHandler(toolName, originalHandler)` so every tool call emits `request.log.info({ toolName, duration, userId, username }, 'Tool invocation: {toolName}')`. If Phase 1's createMcpServer is not yet available, add a TODO comment documenting that tool handlers must be wrapped with wrapToolHandler.

   c. Registers `GET /mcp/events` route (SSE endpoint):
      - Same auth protection via the scoped preHandler
      - Include correlation ID in the SSE stream: send an SSE comment line `: correlationId={request.id}` at the start of the stream
      - Wrap transport handling in requestContext.run() same as POST /mcp

   **Critical: In async Fastify hooks, ALWAYS `return reply` after `reply.send()` to stop hook chain execution (Pitfall 3 from research).**

   **Critical: Use `preHandler` (not `onRequest`) for auth -- body parsing must complete first for POST /mcp (Pitfall 1).**
