  <name>Task 1: Create RFC 6750 error mapping module with tests</name>
  <files>src/auth-errors.ts, tests/auth-errors.test.ts</files>
  <behavior>
    - mapJoseErrorToRfc6750(JWTExpired instance, metadataUrl) returns { statusCode: 401, error: "invalid_token", errorDescription: "Token expired", wwwAuthenticate with resource_metadata }
    - mapJoseErrorToRfc6750(JWTClaimValidationFailed with claim="aud", metadataUrl) returns { errorDescription: "Invalid audience" }
    - mapJoseErrorToRfc6750(JWTClaimValidationFailed with claim="iss", metadataUrl) returns { errorDescription: "Invalid issuer" }
    - mapJoseErrorToRfc6750(JWTClaimValidationFailed with claim="nbf", metadataUrl) returns { errorDescription: "Claim validation failed: nbf" }
    - mapJoseErrorToRfc6750(JWSSignatureVerificationFailed, metadataUrl) returns { statusCode: 401, errorDescription: "Signature verification failed" }
    - mapJoseErrorToRfc6750(JWKSNoMatchingKey, metadataUrl) returns { statusCode: 401, errorDescription: "No matching signing key" }
    - mapJoseErrorToRfc6750(JWKSTimeout, metadataUrl) returns { statusCode: 503, error: "temporarily_unavailable", errorDescription: "Authorization server unreachable" }
    - mapJoseErrorToRfc6750(unknown error, metadataUrl) returns { statusCode: 401, error: "invalid_token", errorDescription: "Invalid or malformed token" }
    - All 401 responses include resource_metadata in wwwAuthenticate string
    - 503 response does NOT include resource_metadata in wwwAuthenticate
    - Missing token case: mapMissingTokenError(metadataUrl) returns { statusCode: 401, error: "invalid_request", errorDescription: "No access token provided" }
    - Insufficient scope case: mapInsufficientScopeError(metadataUrl, requiredScopes) returns { statusCode: 403, error: "insufficient_scope", wwwAuthenticate with scope parameter }
  </behavior>
  <action>
1. Create `src/auth-errors.ts`:

   Export the `Rfc6750Error` interface:
   ```typescript
   export interface Rfc6750Error {
     statusCode: number;
     error: string;
     errorDescription: string;
     wwwAuthenticate: string;
   }
   ```

   Export `mapJoseErrorToRfc6750(err: unknown, resourceMetadataUrl: string): Rfc6750Error`:
   - Use `instanceof` checks against jose error classes (NOT string matching)
   - Map each jose error type per the behavior spec above
   - Use realm="mcp" in all WWW-Authenticate values
   - Include `resource_metadata` URL in WWW-Authenticate for 401 responses
   - JWKSTimeout maps to 503 with error="temporarily_unavailable" (per Phase 4 CONTEXT decision: JWKS fetch failure = 503, not 401)
   - Fallback for unknown errors: 401, "invalid_token", "Invalid or malformed token"

   Export `mapMissingTokenError(resourceMetadataUrl: string): Rfc6750Error`:
   - Returns 401, error="invalid_request", "No access token provided"
   - WWW-Authenticate: `Bearer realm="mcp", resource_metadata="{url}"` (no error field per RFC 6750 when no token at all -- actually include error for consistency)

   Export `mapInsufficientScopeError(resourceMetadataUrl: string, requiredScopes: string[]): Rfc6750Error`:
   - Returns 403, error="insufficient_scope"
   - WWW-Authenticate includes `scope` parameter with space-separated scope list

   All WWW-Authenticate values must follow RFC 6750 quoting rules: use simple ASCII strings for error_description, no special characters.

   - Import jose error classes and construct instances for each test

   - Verify statusCode, error, errorDescription, and wwwAuthenticate format
   - Verify resource_metadata URL appears in wwwAuthenticate for 401 but not 503
   - Use vitest describe/it/expect pattern

  <name>Task 2: Create request context and logging configuration modules</name>
  <files>src/request-context.ts, src/logging.ts</files>
  <action>
1. Create `src/request-context.ts`:

   ```typescript
   import { AsyncLocalStorage } from "node:async_hooks";
   import type { Logger } from "pino";

   export interface RequestContext {
     correlationId: string;
     userId?: string;
     username?: string;
     log: Logger;
   }

   export const requestContext = new AsyncLocalStorage<RequestContext>();
   ```

   This module provides the bridge between Fastify request context and MCP SDK tool handlers. Plan 02 will call `requestContext.run()` in the POST /mcp route handler, and tool handlers will call `requestContext.getStore()` to access correlation ID, user identity, and the enriched logger.

2. Create `src/logging.ts`:

   Export `UUID_REGEX` constant: `/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i`

   Export `buildLoggerConfig()` function that returns a Fastify server options object (partial) with:
   - `logger`: `{ level: "info" }` -- structured JSON pino output (Fastify default)
   - `requestIdHeader`: `false` -- disable automatic header acceptance to prevent log injection (Pitfall 2 from research)
   - `requestIdLogLabel`: `"correlationId"` -- pino field name in log output
   - `genReqId`: function that reads `req.headers["x-request-id"]`, validates against UUID_REGEX, returns client value if valid or generates new UUID v4 via `uuidv4()` if invalid/missing
   - The returned object should be spread into Fastify constructor options

   Import `v4 as uuidv4` from `uuid` for the genReqId function.

   Do NOT export a complete Fastify instance -- only export the configuration options so server.ts can merge them with existing config.

   Both files must use ESM (`import`/`export`) and include TypeScript types.
