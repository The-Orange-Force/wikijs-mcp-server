<feature>
  <name>GET /authorize — OAuth authorization redirect proxy</name>
  <files>src/routes/oauth-proxy.ts, tests/authorize.test.ts</files>
  <behavior>
    AUTHZ-01 behaviors:
    - GET /authorize with valid client_id, redirect_uri, response_type=code, scope=wikijs:read returns 302
    - Redirect Location points to https://login.microsoftonline.com/{tenantId}/oauth2/v2.0/authorize
    - Redirect URL scope param contains api://{clientId}/wikijs:read openid offline_access (mapped + appended)
    - Multiple scopes (wikijs:read wikijs:write) both get prefixed with api://{clientId}/
    - Scope with existing openid/offline_access deduplicates (appears exactly once each, at end)
    - No scope parameter: redirect URL contains just openid offline_access
    - Empty scope string: same as no scope — just openid offline_access
    - Resource parameter in query is stripped from redirect URL
    - Unknown scopes pass through to Azure AD unchanged (e.g. custom:scope stays as custom:scope)

    AUTHZ-02 behaviors:
    - redirect_uri passed through unchanged in redirect URL
    - state passed through unchanged in redirect URL
    - code_challenge passed through unchanged in redirect URL
    - code_challenge_method passed through unchanged in redirect URL
    - nonce, prompt, login_hint passed through when present
    - Unknown query parameters (e.g. foo=bar) are dropped (not in redirect URL)
    - Dropped params logged at debug level

    Validation behaviors (error handling per user decisions):
    - Missing client_id: HTTP 400 JSON {"error": "invalid_request", "error_description": "missing required parameter: client_id"}
    - Wrong client_id (not matching AZURE_CLIENT_ID): HTTP 400 JSON {"error": "invalid_client", "error_description": "unknown client_id"}, warn-level log with received client_id
    - Missing redirect_uri: HTTP 400 JSON {"error": "invalid_request", "error_description": "missing required parameter: redirect_uri"}
    - Missing response_type (redirect_uri present): 302 redirect to redirect_uri with ?error=invalid_request&error_description=missing required parameter: response_type (&state if provided)
    - response_type != "code" (redirect_uri present): 302 redirect to redirect_uri with ?error=unsupported_response_type&error_description=response_type must be 'code' (&state if provided)
    - Successful redirect: info-level log
  </behavior>
  <implementation>
    Add GET /authorize route handler inside the existing oauthProxyRoutes plugin in src/routes/oauth-proxy.ts.

    The handler follows this sequence (per locked user decisions on validation order):

    1. Parse query params using Zod schema with .passthrough() — all fields optional strings, validates presence/type only. Use z.object with all OAuth params as z.string().optional(). Business validation happens in handler logic, not Zod.

    2. Validate client_id (pre-redirect — JSON errors):
       - Missing: return reply.code(400).send({error: "invalid_request", error_description: "missing required parameter: client_id"})
       - Mismatch with appConfig.azure.clientId: request.log.warn({receivedClientId}, "client_id mismatch"), return reply.code(400).send({error: "invalid_client", error_description: "unknown client_id"})

    3. Validate redirect_uri (pre-redirect — JSON errors):
       - Missing: return reply.code(400).send({error: "invalid_request", error_description: "missing required parameter: redirect_uri"})

    4. Helper function for redirect errors (used from this point on):
       ```
       function redirectError(redirectUri, error, description, state?) {
         const url = new URL(redirectUri);
         url.searchParams.set("error", error);
         url.searchParams.set("error_description", description);
         if (state) url.searchParams.set("state", state);
         return reply.redirect(url.toString());
       }
       ```

    5. Validate response_type (post-redirect — redirect errors):
       - Missing: redirect error "invalid_request" / "missing required parameter: response_type"
       - Not "code": redirect error "unsupported_response_type" / "response_type must be 'code'"

    6. Build outbound scope string:
       - Parse incoming scope: scope ? scope.split(" ").filter(Boolean) : []
       - Map via mapScopes(parsedScopes, appConfig.azure.clientId)
       - Filter out any existing openid/offline_access
       - Append openid and offline_access at end
       - Join with space

    7. Build outbound URL using whitelist + URLSearchParams:
       - Define ALLOWED_PARAMS Set: client_id, redirect_uri, response_type, scope, state, code_challenge, code_challenge_method, nonce, prompt, login_hint
       - Create new URLSearchParams()
       - Iterate query entries: if in ALLOWED_PARAMS, set on outbound params; else log at debug level
       - Override scope with final mapped scope string
       - Do NOT include resource (already excluded by whitelist, but the stripResourceParam is available if needed)
       - Construct URL: `${buildAzureEndpoints(appConfig.azure.tenantId).authorize}?${outbound.toString()}`

    8. Log successful redirect at info level, return reply.redirect(azureUrl)

    IMPORTANT implementation notes:
    - Always `return reply.redirect(...)` or `return reply.code(...).send(...)` to prevent double-response (Pitfall 5 from research)
    - Use URLSearchParams for URL construction, never string concatenation (Pitfall 1)
    - Split scope on space with .filter(Boolean) to handle empty string edge case (Pitfall 6)
    - Fastify request.query provides decoded values — do not double-encode (Pitfall 1)
    - Include state in redirect-based error responses when client provided it (Pitfall 3, RFC 6749 section 4.1.2.1)
  </implementation>
</feature>
