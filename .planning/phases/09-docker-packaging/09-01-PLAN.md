Create `Dockerfile` at the project root. Use two stages: `builder` and `runtime`. Both stages use `node:20-slim` as the base (not Alpine — @azure/msal-node has documented musl libc compatibility issues with Alpine).

**Builder stage:**
- `FROM node:20-slim AS builder`
- `WORKDIR /app`
- Copy `package*.json` first, then run `npm ci` (all dependencies, including devDependencies — needed for tsc)
- Copy `tsconfig.json` and `src/`
- Run `npm run build` (compiles TypeScript to dist/)
- Strip source maps and type declarations that tsc emits but are not needed at runtime:
  `RUN find dist/ -name '*.map' -o -name '*.d.ts' | xargs rm -f`
  (Note: .dockerignore cannot do this — it only filters the host build context, not files generated inside the builder stage)

**Runtime stage:**
- `FROM node:20-slim AS runtime`
- `WORKDIR /app`
- Copy `package*.json` first, then run `npm ci --omit=dev` (production dependencies only — use `--omit=dev` not the deprecated `--only=production`)
- `COPY --from=builder /app/dist ./dist/` (copy stripped dist from builder)
- `COPY lib/ ./lib/` (copy STDIO stub — lib/mcp_wikijs_stdin.js)
- `USER node` — drop to non-root user before CMD. `node:20-slim` pre-creates the `node` system user (uid 1000). No chown needed — dist/ is read-only at runtime and Pino logs to stdout.
- HEALTHCHECK: use `node -e` with Node's built-in `http` module — curl and wget are NOT present in node:20-slim (they are purged during the Node.js installation process to keep the image minimal):
  ```
  HEALTHCHECK --start-period=30s --interval=30s --timeout=10s --retries=3 \
    CMD node -e "require('http').get('http://localhost:' + (process.env.PORT || 8000) + '/health', r => process.exit(r.statusCode === 200 ? 0 : 1)).on('error', () => process.exit(1))"
  ```
  The fallback port is `8000` — that is what `src/config.ts` defaults to (`z.string().default("8000")`). Do not hardcode 3200.
- `CMD ["node", "dist/server.js"]` — not `npm start`. npm adds shell/process indirection and SIGTERM from `docker stop` reaches npm instead of Node, causing a hard kill.

<task type="auto">
  <name>Task 2: Create docker-compose.yml</name>
  <files>docker-compose.yml</files>
  <action>
Create `docker-compose.yml` at the project root.

Service definition (`wikijs-mcp-server`):
- `build: .` (builds from Dockerfile in project root)
- `image: wikijs-mcp:latest`
- `container_name: wikijs-mcp-server` (Caddy config uses this name for DNS resolution: `reverse_proxy wikijs-mcp-server:PORT`)
- `restart: unless-stopped` (automatic recovery from failures)
- `env_file: .env` (operator supplies all env vars via .env — never hardcode secrets)
- `networks: [caddy_net]`
- No `ports:` key at all — Caddy accesses the container via the caddy_net network only. Publishing a port would expose the JWT-protected MCP endpoint over plain HTTP, bypassing Caddy's TLS termination.

Networks block:
```yaml
networks:
  caddy_net:
    external: true
    name: caddy_net
```
`external: true` means Docker Compose joins a pre-existing network rather than creating one. `caddy_net` is the confirmed network name on the target host (matches actual Caddy Docker network).

Include a comment block in the docker-compose.yml explaining:
1. No ports: mapping is intentional — Caddy accesses via caddy_net by service name
2. Caddyfile snippet showing how to wire Caddy:
   ```
   # your-domain.example.com {
   #   reverse_proxy wikijs-mcp-server:8000
   # }
   ```
   (Use PORT placeholder in the comment — the actual port comes from the .env file)
3. Operator pre-requisite: `caddy_net` must exist before `docker compose up`. If absent: `docker network create caddy_net` (or start the Caddy compose stack first)
4. Note about WIKIJS_BASE_URL: must be the actual host IP or hostname, not `localhost` — inside a container, `localhost` resolves to the container itself, not the host
  </action>
